# AI-Agent Master Guidelines (Source of Truth)

This file is the single source of truth for all AI agents (Gemini, Claude, GPT/Codex) maintaining this repository.

## 🚀 Agentic Bootstrap (Creating a New Project)
If you are an AI agent tasked with initializing a new project from this template, follow these steps immediately after cloning:

1.  **Environment Sync**: Run `cd frontend && npm install` to initialize the frontend environment.
2.  **Config Setup**: Run `cp .env.example .env` and `cp .env.example backend/.env` to establish default configuration.
3.  **Dependency Alignment**:
    - The backend uses **workspace dependency management** (`[workspace.dependencies]` in `backend/Cargo.toml`). All shared dependency versions are centralized there.
    - Always check `backend/Cargo.toml` and ensure `home`, `getrandom`, `uuid`, and `clap` are pinned as per the existing lockfile/manifest.
    - When adding a new dependency used by multiple crates, add it to `[workspace.dependencies]` first, then reference it with `{ workspace = true }` in member crates.
4.  **Initial Type Export**: Run `./scripts/export-types.sh` to ensure the frontend types are in sync with the backend.
5.  **Baseline Verification**: Run `make verify` to ensure the template is healthy before adding new features.

## 🏗 Architectural Mandates (Non-Negotiable)
1.  **Domain Purity**: `backend/crates/domain` MUST NOT depend on any other internal crate. Never import `infra`, `api`, or `cli` into `domain`.
2.  **Identity Control**: All domain IDs (e.g., `UserId`) must be **Newtypes** with private inner fields and generated by the backend (Domain/Application layer). Never use raw `Strings`. Provide `as_str()` accessors and `Display` impls.
3.  **Value Object Validation**: Value objects (e.g., `Email`, `Username`) MUST enforce invariants at construction via `new()` constructors that return `Result<Self, DomainError>`. Use `from_persisted()` only for reconstitution from storage where data was already validated on the way in.
4.  **Error Protocol**: Every fallible domain operation MUST return a `DomainError` enum. Use domain-level vocabulary for error variants (e.g., `RepositoryError`, not `DatabaseError`). Infrastructure concepts must not leak into the domain.
5.  **Type Bridge**: Use `ts-rs` with `#[ts(export)]` for all DTOs and Domain structs shared with the frontend.
6.  **Observability**: Use `#[instrument]` from the `tracing` crate for all Use Case and Repository methods.
7.  **Native Async Traits**: Use Rust's native async in traits (stable since Rust 1.75). Do NOT use the `async-trait` crate.
8.  **Axum State**: Use `axum::extract::State` (not `Extension`) for sharing application state in handlers. `State` provides compile-time guarantees.
9.  **Workspace Dependencies**: All shared dependency versions are managed centrally in `backend/Cargo.toml` under `[workspace.dependencies]`. Member crates reference them with `{ workspace = true }`.

## 🤖 Agentic Workflow & Quality of Life
- **Verification**: Always run `make verify` (or `scripts/verify.sh`) before concluding a task.
- **CLI Adapter**: Execute backend use cases directly via the terminal using `make admin ARGS="..."`. This is preferred over manual HTTP testing.
- **Port Mapping**: Frontend (3000), Backend API (3001).
- **CORS**: The backend reads `CORS_ORIGIN` from environment (defaults to `http://localhost:3000`). Set this in `.env` to match your frontend origin.

## 🛠 Project Lifecycle
- **Build**: `cd backend && cargo build`
- **Test**: `make verify`
- **Migrations**: `cd backend && cargo sqlx migrate run`
- **Dev Mode**: `make dev` (Starts all services with auto-cleanup)
